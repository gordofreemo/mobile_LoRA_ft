# Research Project — On-Device LLM Personalization via PEFT

## What this project is

Fine-tuning **SmolLM3-3B** for personalized instruction following using a two-stage LoRA pipeline:

1. **Cluster side (current phase):** Train a Task-LoRA adapter using synthetic preference-conditional data generated by a large teacher model. This teaches the model to follow user preference cues in the system prompt across multiple task domains.
2. **On-device side (future phase, pending hardware):** Train a User-LoRA adapter on top of the Task-LoRA using a specific user's on-device interaction history.

**All current work is cluster-side only.** Do not implement or plan for on-device training unless explicitly asked.

---

## Research questions being answered (Phase 1)

1. Does fine-tuning on LaMP improve over zero-shot at all for a 3B model?
2. Does adding synthetic preference-conditional data on top of LaMP training improve further?
3. Does a **general** Task-LoRA (all domains) perform comparably to **domain-specific** Task-LoRAs?

These are answered by the ablation comparison chain below.

---

## Ablation conditions

| Condition | Training data | What it isolates |
|---|---|---|
| **Baseline** | None (zero-shot) | Raw SmolLM3-3B capability |
| **A1-lamp** | LaMP training splits only | Does fine-tuning on LaMP help at all? |
| **A1-full** | LaMP splits + synthetic preference-conditional data (all domains) | Does synthetic data add value beyond LaMP alone? |
| **A2** | Domain-specific LaMP split + domain synthetic data (3 runs) | Does domain specialization add further value? |

**Critical comparison:** A1-lamp → A1-full. If A1-full ≈ A1-lamp, the synthetic data pipeline adds no value and should be dropped.

Conditions B and C (on-device User-LoRA) are **out of scope for this phase**.

---

## Model

- **Base:** `HuggingFaceTB/SmolLM3-3B`
- **Weights:** bf16, frozen throughout all training
- **Framework:** HuggingFace Transformers + PEFT

---

## Training method — SFT only, no KD

**Do not use knowledge distillation.** CE loss only:

```
Loss = CrossEntropy(Task-LoRA outputs, ground truth)
```

The teacher model (Qwen3-30B or GPT-4o) is used **only for offline synthetic data generation**, never loaded alongside the student during training.

**Do not** co-load teacher and student. **Do not** compute KL divergence loss. **Do not** modify base model weights.

---

## Task-LoRA config

```python
from peft import LoraConfig

task_lora = LoraConfig(
    r=64,
    lora_alpha=128,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
```

- **A1-lamp:** one training run, LaMP splits only (no synthetic data)
- **A1-full:** one training run, LaMP splits + synthetic data mixed
- **A2:** three independent training runs (one per domain), same config each

---

## Training setup

```
Optimizer:    AdamW, lr=3e-4, cosine decay, warmup 3%
Batch:        per-device bs=4, gradient accumulation=8 (effective bs=32)
Epochs:       2–3
Checkpoint:   every 500 steps
Logging:      W&B — CE loss, LaMP metrics per domain, BFCL regression score
```

## Datasets

### LaMP training splits (primary)
Download from the LaMP benchmark. Use subtasks: **LaMP-3, LaMP-4, LaMP-6, LaMP-7**.
- LaMP-3: personalized product rating prediction
- LaMP-4: personalized news headline generation
- LaMP-6: personalized email subject generation
- LaMP-7: personalized tweet paraphrasing

### Synthetic preference-conditional data
Generated offline by the teacher model. Format: JSONL with fields `system`, `user`, `assistant`. The `system` field contains a synthetic user profile; the `assistant` field is a preference-compliant completion.

Generated using contrastive pairs — same user request, two different user profiles, two different outputs. Profiles span all three domains: tool-calling, email/text drafting, recommendations.

**Data mix (A1-full and A2 only):** ~60% LaMP training data + ~40% synthetic preference-conditional data. A1-lamp uses LaMP splits only — no synthetic data.

**Train/val/test split must have no user profile leakage** — a given synthetic profile should appear in only one split.

### Domain-specific corpora (for A2 only)

| Domain | Sources |
|---|---|
| Tool-calling | `Salesforce/xlam-function-calling-60k`, `glaiveai/glaive-function-calling-v2` (filtered) |
| Email / text drafting | LaMP-6 training split, Enron email corpus (public) |
| Recommendations | LaMP-3 training split, Yelp open dataset (public) |

Data mix per A2 run: ~60% domain corpus + ~40% domain-specific synthetic data.

## Evaluation

Run on LaMP-3, LaMP-4, LaMP-6, LaMP-7 test splits.

| Task | Metric |
|---|---|
| LaMP-3 (rating prediction) | Accuracy |
| LaMP-4 (headline generation) | ROUGE-1 |
| LaMP-6 (email subject) | ROUGE-1 |
| LaMP-7 (tweet paraphrase) | ROUGE-1 |

Also run **BFCL regression** before and after each Task-LoRA training run. Target: score stays ≥ 90 (baseline is 92.3). Sanity check only — improvement on BFCL is not a goal.

**Key comparison chain:**
1. Baseline → A1-lamp: does fine-tuning help at all?
2. A1-lamp → A1-full: does synthetic data add value? If not, cut the data generation work.
3. A1-full → A2: does domain specialization add value beyond the general adapter?

## Hard constraints

- **Never modify base model weights.** LoRA only; base is always frozen.
- **No KD loss.** CE only.
- **No co-loading teacher and student.** Teacher is offline data generation only.
- **No on-device / mobile code** in this phase. ExecuTorch, Core ML, and iOS-specific code are out of scope.
- **Reproducibility first.** Every training run must be launchable from one CLI command with a fixed seed. Log the full command in the experiment log entry.
- **No profile leakage between splits.** Validate this explicitly before training.
