# 2026-05-31 — A1-lamp Task-LoRA training

## Hypothesis

Fine-tuning SmolLM3-3B on the LaMP training splits (LaMP-3/4/7) with a
small LoRA adapter — and the BM25-retrieved profile pinned in the system
prompt at *both* train and inference time — improves over the zero-shot
profile-baseline on the same dev sets:

| Task    | Zero-shot profile baseline | A1-lamp target |
|---------|----------------------------|----------------|
| LaMP-3  | acc 0.6924                 | > 0.6924       |
| LaMP-4  | rouge1 0.1615              | > 0.1615       |
| LaMP-7  | rouge1 0.4352              | > 0.4352       |

BFCL AST overall must stay ≥ 0.78 (catastrophic-forgetting guard; current
bare-model is 0.8078).

The deployment-tax tradeoff (profile always in system) was accepted up
front; the open hypothesis is that Phase-2 on-device User-LoRA can later
absorb the profile into adapter weights and let us drop it at inference.
See open-decision #1 in CLAUDE.md.

## Setup

**Command:**
```
condor_submit condor/train.sub
# or, interactive smoke:
python train/train.py --limit 64 --max_steps 4 --no-wandb
```

**Config:** `train/config/a1_lamp.json` (full snapshot saved into
`train/checkpoints/a1_lamp_seed0/train_meta.json` at run end).

**Data:** `data/lamp_train_mixed_bm25k4.jsonl` — 42,964 examples,
interleaved-and-shuffled LaMP-3 (20,000) + LaMP-4 (12,527) + LaMP-7 (10,437),
BM25 k=4 profile already in the `system` field. Built from commit `<TBD>`
via `condor_submit condor/build_dataset.sub` (job 163355, 7m 45s).

**Seed:** 0 (matches eval seeds; tokenized order is the deterministic
shuffle from `build_dataset.py` seed 0).

### Hyperparameter rationale

| Knob | Value | Source |
|---|---|---|
| LoRA r | 4 | OPPU (arXiv:2402.04401); on-device cost scales linearly in r |
| LoRA α | 8 | α/r = 2 (Hu et al. 2021) |
| LoRA dropout | 0.05 | within LoRA paper's 0.0-0.1 range; middle-of-the-road |
| target_modules | q/k/v/o + gate/up/down | QLoRA (Dettmers et al. 2023): all-linear > attention-only |
| Optimizer | adamw_torch | Loshchilov & Hutter 2017; HF Trainer LLM-SFT default |
| LR | 3e-4 | LoRA-typical 1e-4–5e-4 for ~3B; HF PEFT tutorial default |
| weight_decay | 0.0 | Biderman et al. 2024: WD can hurt low-rank adapters |
| Scheduler | cosine + warmup 0.03 | LLaMA recipe (Touvron 2023) |
| Effective batch | 32 (4 × 8 accum) | matches CLAUDE.md spec; job submitted with `gpus_minimum_memory=32000` so the 16 GB loki V100 (where bs=4 OOMs at logits ≈ 2 GiB) is excluded |
| Epochs | 2 | LoRA SFT typically 1-3; >3 overfits per Raschka empirical sweeps |
| max_seq_length | 2048 | LaMP+k=4 profile fits well under 2K; matches eval generation context |
| max_grad_norm | 1.0 | standard transformer training clip |
| Precision | bf16 | CLAUDE.md; base frozen in bf16 |
| Gradient checkpointing | true | memory/compute tradeoff for 3B+ LoRA |

### Train/eval consistency contract

- Role layout (`system` = retrieved profile, `user` = LaMP `input`,
  empty-system → no system message at all) matches
  `eval/eval_lamp.build_messages` exactly.
- Chat template applied with `enable_thinking=False` in both train and
  eval — SmolLM3's optional `<think>` block stays out of both.
- BM25 retrieval (k=4) is locked in `build_dataset.py` and
  `eval/eval_lamp.py` via shared duplicated code (commented intentional).
