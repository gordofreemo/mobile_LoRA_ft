# 2026-05-31 — A1-lamp Task-LoRA training

Fine-tuning **SmolLM3-3B** with a small (r=4) Task-LoRA on the LaMP training corpus (BM25-retrieved profile in `system`, LaMP-{3,4,7} mixed). Pipeline written 2026-05-31; training run executed 2026-06-01; evaluation completed 2026-06-02.

---

## TL;DR

A1-lamp produces **clear personalization gains on every LaMP task** but **regresses tool-calling capability** on BFCL.

- LaMP-3 accuracy: $0.689 \rightarrow 0.800$  ($\Delta = +0.111$, **+16.1% relative**)
- LaMP-4 ROUGE-1: $0.162 \rightarrow 0.224$  ($\Delta = +0.062$, **+38.6% relative**)
- LaMP-7 ROUGE-1: $0.435 \rightarrow 0.558$  ($\Delta = +0.123$, **+28.2% relative**)
- BFCL AST overall: $0.808 \rightarrow 0.699$  ($\Delta = -0.109$, **−13.5% relative**)

Research question **Q1 — "Does fine-tuning on LaMP improve over zero-shot for a 3B model?" — is answered: yes, decisively.** Q2 (does synthetic preference-conditional data add value over LaMP alone?) remains open.

---

## Hypothesis

Fine-tuning SmolLM3-3B on the LaMP training splits (LaMP-3/4/7) with a small LoRA adapter — and the BM25-retrieved profile pinned in the system prompt at *both* train and inference time — improves over the zero-shot profile baseline on the same dev sets:

| Task    | Zero-shot profile baseline | A1-lamp target | Actual |
|---------|----------------------------|----------------|--------|
| LaMP-3  | acc 0.6924                 | > 0.6924       | **0.8004** ✅ |
| LaMP-4  | rouge1 0.1615              | > 0.1615       | **0.2238** ✅ |
| LaMP-7  | rouge1 0.4352              | > 0.4352       | **0.5581** ✅ |

BFCL AST overall must stay ≥ 0.78 (catastrophic-forgetting guard; pre-training bare-model is 0.8078). **Actual: 0.6991** ❌ — see Result for the regression analysis.

The deployment-tax tradeoff (profile always in `system`) was accepted up front; the open hypothesis is that Phase-2 on-device User-LoRA can later absorb the profile into adapter weights and let us drop it at inference. See open-decision #1 in CLAUDE.md.

### Data integrity — no train/dev leakage

LaMP uses a user-based split: each user appears in exactly one of {train, dev, test}. Verified empirically on our local copy of the data before reporting results.

**Question-ID disjointness** (full sets):

| Task | Train $n$ | Dev $n$ | $\lvert \text{train} \cap \text{dev} \rvert$ |
|---|---:|---:|---:|
| LaMP-3 | 20,000 | 2,500 | **0** |
| LaMP-4 | 12,527 | 1,925 | **0** |
| LaMP-7 | 10,437 | 1,500 | **0** |

**Profile-entry-ID disjointness** (sample of 100 questions per split — each LaMP question carries that user's history as `profile`, so overlapping profile IDs across train and dev would indicate a shared user):

| Task | Train pool | Dev pool | Overlap |
|---|---:|---:|---:|
| LaMP-3 | 16,799 | 18,657 | **0** |
| LaMP-4 | 12,676 | 10,062 | **0** |
| LaMP-7 | 1,782  | 1,755  | **0** |

A 35k-mixed-IDs sample yielding zero overlap is consistent with the user-based protocol (`data/download_lamp.py:84`). In a time-based split we would expect dense overlap. Conclusion: **dev users (and their profile content) are unseen by the model during training**.

### Hyperparameter rationale

| Knob | Value | Source |
|---|---|---|
| LoRA $r$ | 4 | OPPU (arXiv:2402.04401); on-device cost scales linearly in $r$ |
| LoRA $\alpha$ | 8 | $\alpha/r = 2$ (Hu et al. 2021) |
| LoRA dropout | 0.05 | within LoRA paper's 0.0–0.1 range; middle-of-the-road |
| target_modules | $\{q,k,v,o\}_\text{proj} \cup \{\text{gate},\text{up},\text{down}\}_\text{proj}$ | QLoRA (Dettmers et al. 2023): all-linear > attention-only |
| Optimizer | adamw_torch | Loshchilov & Hutter 2017; HF Trainer LLM-SFT default |
| Learning rate | $3\times 10^{-4}$ | LoRA-typical $10^{-4}$–$5\times 10^{-4}$ for ~3B; HF PEFT tutorial default |
| weight_decay | 0.0 | Biderman et al. 2024: WD can hurt low-rank adapters |
| Scheduler | cosine + warmup $0.03$ | LLaMA recipe (Touvron 2023) |
| Effective batch | 32 (4 × 8 accum) | matches CLAUDE.md spec; job submitted with `gpus_minimum_memory=32000` so the 16 GB loki V100 (where bs=4 OOMs at logits $\approx$ 2 GiB) is excluded |
| Epochs | 2 | LoRA SFT typically 1–3; >3 overfits per Raschka empirical sweeps |
| max_seq_length | 2048 | LaMP + $k=4$ profile fits well under 2K; matches eval generation context |
| max_grad_norm | 1.0 | standard transformer training clip |
| Precision | bf16 | CLAUDE.md; base frozen in bf16 |
| Gradient checkpointing | true | memory/compute tradeoff for 3B+ LoRA |

### Train/eval consistency contract

- Role layout (`system` = retrieved profile, `user` = LaMP `input`, empty-system → no system message at all) matches `eval/eval_lamp.build_messages` exactly.
- Chat template applied with `enable_thinking=False` in both train and eval — SmolLM3's optional `<think>` block stays out of both.
- BM25 retrieval ($k=4$) is locked in `build_dataset.py` and `eval/eval_lamp.py` via shared duplicated code (commented intentional).

## Result

### Training run

- **Train wall time:** 5h 08m (18,447 s; 2,682 optimizer steps; 4.65 samples/s; 1.20 PFLOPs)
- **Job:** Condor cluster `163392.0`, A100-PCIE-40GB @ `gna.hpc.uni-saarland.de`
- **Adapter:** `train/checkpoints/a1_lamp_seed0/final/` — 30 MB safetensors, r=4 LoRA across 7 modules
- **Metric files:** `train/checkpoints/a1_lamp_seed0/{metrics.jsonl, log_history.json, train_meta.json}`
- **Final point-wise loss:** $0.922$ (aggregate $\bar{\mathcal{L}}_{\text{train}} = 1.043$; min $0.844$ at step 1390)

**Loss trajectory** (mean over each window of optimizer steps):

| Window | Mean training loss |
|---|---:|
| 1–100      | 1.354 |
| 100–500    | 1.136 |
| 500–1000   | 1.124 |
| 1000–1500  | 1.066 |
| 1500–2000  | **0.944** |
| 2000–2500  | 0.944 |
| 2500–2682  | 0.935 |

Start loss $1.94 \rightarrow$ end loss $0.92$. Minimum point-wise loss ($0.844$) occurs at **step 1390**, well into epoch 1; the second epoch contributes only ~$0.13$ nats over the first. Grad-norm stays around $0.83$ throughout (one early spike to $1.51$, well-handled by clipping at $\|g\| \le 1$). LR cosine-decays cleanly from $3\times 10^{-4}$ to $9.85\times 10^{-10}$.

### LaMP evaluation (dev split, BM25 $k=4$, greedy)

All three conditions share the same retrieval, formatting, and decoding; only the adapter weights change.

#### LaMP-3 — Personalized rating prediction (Accuracy ↑, MAE ↓)

| Condition | Accuracy | MAE | Parse-fail | Mean prompt tok | Mean gen tok | $n$ |
|---|---:|---:|---:|---:|---:|---:|
| No-profile floor              | 0.4404 | 0.628 | 0.0000 | 258.6 | 2.0 | 2500 |
| Profile baseline (BM25 $k=4$) | 0.6892 | 0.372 | 0.0004 | 741.2 | 2.4 | 2500 |
| **A1-lamp adapter**           | **0.8004** | **0.220** | 0.0000 | 741.2 | 2.0 | 2500 |

$$\Delta_{\text{adapter} - \text{baseline}}^{\text{LaMP-3 acc}} = 0.800 - 0.689 = +0.111$$

The adapter improves not only exact-match accuracy but also **ordinal accuracy** (MAE $0.37 \rightarrow 0.22$) — i.e., when it's wrong, it's wrong by less. Parse-fail rate goes to zero (adapter learned to emit the digit token cleanly).

#### LaMP-4 — Personalized headline generation (ROUGE-1 F1 ↑)

| Condition | ROUGE-1 | Mean prompt tok | Mean gen tok | $n$ |
|---|---:|---:|---:|---:|
| No-profile floor              | 0.1410 | 105.3 | 26.3 | 1925 |
| Profile baseline (BM25 $k=4$) | 0.1615 | 340.6 | 24.8 | 1925 |
| **A1-lamp adapter**           | **0.2238** | 340.6 | **14.4** | 1925 |

$$\Delta_{\text{adapter} - \text{baseline}}^{\text{LaMP-4 rouge1}} = 0.224 - 0.162 = +0.062$$

The 42% reduction in mean generated tokens ($24.8 \rightarrow 14.4$) is an **emergent personalization signal**: the model learned that user headlines tend to be short and now terminates earlier. Lower length on a metric that punishes verbosity contributes to the ROUGE-1 gain.

#### LaMP-7 — Personalized tweet paraphrasing (ROUGE-1 F1 ↑)

| Condition | ROUGE-1 | Mean prompt tok | Mean gen tok | $n$ |
|---|---:|---:|---:|---:|
| No-profile floor              | 0.4139 | 107.6 | 28.2 | 1500 |
| Profile baseline (BM25 $k=4$) | 0.4352 | 226.3 | 27.6 | 1500 |
| **A1-lamp adapter**           | **0.5581** | 226.3 | 25.7 | 1500 |

$$\Delta_{\text{adapter} - \text{baseline}}^{\text{LaMP-7 rouge1}} = 0.558 - 0.435 = +0.123$$

The largest absolute lift of the three tasks. The retrieval channel for the baseline was already strong on LaMP-7 (floor → baseline only $+0.021$), so the adapter is doing real personalization work beyond what BM25 can carry in-context.

### BFCL AST regression (catastrophic-forgetting check)

CLAUDE.md mandates BFCL before/after every Task-LoRA run with a target of "stays ≥ 90" (reference 92.3). Our pre-training baseline already sits below that at 80.78 due to known Java/JS parser issues (see CLAUDE.md, "Open design decisions" §3). The adapter regresses further:

| Category | $n$ | Baseline acc | A1-lamp acc | $\Delta$ (pp) |
|---|---:|---:|---:|---:|
| simple_python      | 400 | 0.9425 | 0.8625 | $-8.00$ |
| simple_java        | 100 | 0.2500 | 0.1800 | $-7.00$ |
| simple_javascript  |  50 | 0.3400 | 0.2600 | $-8.00$ |
| parallel           | 200 | 0.8400 | 0.7400 | $-10.00$ |
| multiple           | 200 | 0.9250 | 0.6850 | **$-24.00$** |
| parallel_multiple  | 200 | 0.7850 | 0.7150 | $-7.00$ |
| **Overall (weighted)** | **1150** | **0.8078** | **0.6991** | **$-10.87$** |

Python-only subset (simple_python + parallel + multiple + parallel_multiple, $n=1000$):

$$\text{acc}^{\text{Py}}_{\text{baseline}} = 0.887 \quad\longrightarrow\quad \text{acc}^{\text{Py}}_{\text{adapter}} = 0.770 \quad (\Delta = -0.117)$$

The adapter's largest individual loss is on the `multiple` category (choose-the-right-tool-from-a-menu): −24 pp. Smaller, broadly comparable losses (~7–10 pp) on every other category, indicating the regression is **not** a single failure mode — it's a diffuse degradation of the format / output-control behavior the base model used to have. Per-error breakdown also shows a sharp increase in `simple_function_checker:wrong_count` (16 → 69) and `parallel_function_checker:wrong_count` (41 → 66) — the adapter emits too few calls when many are required, consistent with LaMP's strong "terminate early" prior bleeding into tool-call generation.

This is **catastrophic forgetting**, and it's the cost of using all $2{,}682 \times 32 \approx 86\text{k}$ effective updates on personalization data with no retention pressure on tool-calling. It is a real result to report, not a metric artifact: the model loaded the LaMP-style "be short, emit literal answer" prior so strongly that it overrode its native function-calling format adherence.

### Caveats

1. **BFCL scorer placeholder.** SmolLM3-3B isn't in BFCL's `MODEL_CONFIG_MAPPING`, so both runs use `meta-llama/Llama-3.1-8B-Instruct` as a neutral placeholder for output convention lookup. This is captured in every result record (`scorer_model_name_placeholder`) and is held constant across baseline and adapter, so the $\Delta$ is unaffected.
2. **`irrelevance` category is skipped.** The BFCL data file `possible_answer/BFCL_v4_irrelevance.json` isn't shipped; both baseline and adapter omit it, so the comparison is fair but the overall number is on a 6-category subset.
3. **Minimum-training-loss checkpoint is at step 1390**, not step 2682. The final adapter is slightly above its own training-loss minimum (~0.08 nats). Whether `checkpoint-1500` evaluates better on LaMP and/or BFCL remains untested.

## Conclusion

| Q | Status |
|---|---|
| **Q1** — Does FT on LaMP improve over zero-shot? | **Answered: YES.** Strong gains on all three tasks (8.0–17.7 pp absolute or 16–39% relative over the BM25-baseline). |
| **Q2** — Does synthetic preference-conditional data add value beyond LaMP alone? | **Open.** Requires A1-full run, which requires synthetic data generation, which has not been started. |
| **Q3** — Does a general Task-LoRA generalize as well as domain-specific ones? | **Open.** Requires A2 (×3 domain-specific runs). |

The headline LaMP gains validate the general approach. The BFCL regression is the new finding that wasn't anticipated by Q1–Q3 but is highly relevant to the **on-device deployment story** that motivates the project: a personalization adapter that wrecks function-calling is a non-starter for an agentic on-device assistant. Mitigation needs to be in scope.

### Recommended next steps

Two threads, ordered by cost:

1. **Cheap diagnostic — eval `checkpoint-1500` (the minimum-training-loss neighborhood) on both LaMP and BFCL.** Same eval infrastructure, no retraining. If checkpoint-1500 retains most of the LaMP lift but recovers some BFCL ground, we have a "free" Pareto improvement and can declare early stopping the correct response to this experiment. If it looks identical, we know step count isn't the dial.
2. **A1-full preparation — begin synthetic preference-conditional data generation.** This is the gating dependency for answering Q2. The data mix per CLAUDE.md is ~60% LaMP + ~40% synthetic, with the synthetic pool spanning the three domains (tool-calling, email/text drafting, recommendations). The tool-calling slice of the synthetic data is incidentally a candidate **BFCL-retention vehicle** — if A1-full also degrades BFCL, the regression is structural to LaMP-style fine-tuning at this rank/scale; if A1-full degrades BFCL **less**, the broader data distribution is doing retention work and we have a concrete mechanism for the on-device deployment.

If A1-full also degrades BFCL meaningfully, downstream mitigations to consider (out of scope today): smaller $r$, fewer epochs, mixing a small fraction of explicit tool-calling examples into the corpus, or a KL anchor toward the base model on tool-calling-shaped inputs.
